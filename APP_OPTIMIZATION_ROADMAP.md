# 衣柜管家 App 优化 Roadmap

## 1. 文档目标

本文档用于梳理当前 `衣柜管家` App 的主要问题、优化方向、阶段优先级与建议落地顺序，帮助后续迭代从 MVP 走向更稳定、更好用、更易维护的产品版本。

适用范围：

- 产品体验优化
- 页面 UI 与交互优化
- 功能补全
- 性能与稳定性提升
- 数据结构与代码可维护性优化

---

## 2. 当前版本定位

当前项目已经具备完整的 MVP 闭环，包含以下三条主链路：

- 添加衣物
- 搜配生成
- 历史记录回看

现阶段的优点：

- 功能主流程已打通
- 本地数据存储与展示闭环完整
- 图片抠图、搜配生成、历史记录等核心能力已具雏形
- 代码结构已初步按 `Feature + Service + Model` 分层

现阶段的限制：

- 产品可用，但仍偏“演示版 / 初版工具”
- 关键交互存在体验断点
- 部分核心能力可信度不够高
- 数据模型已经出现早期方案遗留
- 测试覆盖不足，后续迭代风险较高

---

## 3. 当前主要问题

### 3.1 产品体验问题

- 衣柜页缺少搜索、排序、多维筛选，单品数量增加后管理效率会快速下降。
- 单品详情页只能删除或标记“今天穿了”，缺少编辑能力。
- 搜配页价值感偏弱，更多是“生成结果”，而不是“帮助做决策”。
- 历史记录页缺少场景语义、备注、复用入口，回看价值有限。

### 3.2 页面 UI 问题

- 当前页面整体能用，但层级感和品牌感较弱。
- 衣柜页、搜配页、历史页之间缺少更鲜明的视觉区分。
- 搜配评分区虽然已支持人工改分，但还可以进一步提升可读性与操作感。
- 添加衣物页的图片区还可以更聚焦“选图、抠图、确认保存版本”的主任务。

### 3.3 交互流程问题

- 添加衣物保存后流程中断，不适合连续录入多件单品。
- 搜配中更换单品后，预览和评分失效虽合理，但目前提示仍不够强。
- 删除、保存、穿着记录等关键操作的成功与失败反馈不够系统。
- 选品器仅支持基础选择，缺少更高效的确认机制。

### 3.4 功能完整性问题

- 缺少单品编辑。
- 缺少搜配备注、使用场景、天气或季节上下文。
- 缺少“再次使用这套搭配”“收藏搭配”等复用能力。
- 缺少针对衣物使用频率的更强统计与提醒能力。

### 3.5 性能与稳定性问题

- 本地图像处理和预览生成在图片数量增多后可能带来内存与主线程压力。
- 列表与详情中存在多处直接图片解码，后期可能影响滚动性能。
- 部分关键失败分支仍以开发期断言为主，不适合长期产品化。

### 3.6 数据结构与可维护性问题

- 评分系统已从“系统分项打分展示”转向“系统默认总分 + 用户最终改分”，但模型里仍保留旧分项字段。
- 某些业务状态直接放在 View 内部，后续页面逻辑继续增长时会增加维护成本。
- Service 与 View 的职责边界还可以进一步清晰化。

### 3.7 测试问题

- 当前测试覆盖集中在评分规则，范围较窄。
- 图片处理、保存流程、快照一致性、评分改分落库等关键流程缺少自动化保护。

---

## 4. 优化方向分类

本项目后续优化建议分为 7 个方向：

1. 产品体验
2. 页面 UI 与视觉表达
3. 交互流程
4. 功能完整性
5. 性能与稳定性
6. 数据结构与代码架构
7. 测试与工程质量

---

## 5. 各方向优化建议

### 5.1 产品体验

目标：让用户在“录入衣物、搭配衣物、回看记录”三个核心场景里更顺手、更愿意持续使用。

建议：

- 为衣柜页增加搜索能力。
- 增加排序方式，例如最近添加、最近穿过、穿着次数少。
- 增加多维筛选，例如颜色、风格、季节。
- 为搜配增加“场景”输入，例如通勤、出游、聚会。
- 为搜配记录增加备注字段，保存用户当时的主观判断。
- 为历史记录增加“再次使用这套”入口，提升复用效率。

### 5.2 页面 UI 与视觉表达

目标：在不破坏现有信息结构的前提下，提高界面质感、层级感和识别度。

建议：

- 优化衣柜首页头部与筛选区层级，让核心信息更聚焦。
- 强化卡片式布局的一致性，统一留白、圆角、阴影与间距规则。
- 提升搜配页步骤感，让“选品 -> 生成 -> 评分 -> 保存”更清晰。
- 优化历史记录列表的信息密度，增强可扫读性。
- 为评分模块建立统一视觉样式，强化“默认分”和“最终分”的关系表达。

### 5.3 交互流程

目标：减少中断、降低误操作，提高高频操作效率。

建议：

- 添加衣物支持“保存并继续添加”。
- 添加衣物页支持更明确的重新选图入口。
- 搜配页在选品变更后增加更直观的“需重新生成”提示。
- 为关键操作增加统一的成功 / 失败反馈机制。
- 在单品选择页增加当前选择状态的强化展示。

### 5.4 功能完整性

目标：补齐用户实际使用中最容易感知缺失的能力。

建议：

- 增加单品编辑页或复用添加页做编辑模式。
- 增加搭配备注与使用场景。
- 增加搭配收藏能力。
- 增加“常穿 / 待穿 / 很久没穿”标签体系。
- 增加历史搭配快速复用或继续编辑功能。

### 5.5 性能与稳定性

目标：在数据量和图片量增加后，保持体验稳定。

建议：

- 保存图片时统一做尺寸与质量压缩策略。
- 生成适合列表展示的小图版本，减少重复解码成本。
- 为本地图片读取加入轻量缓存。
- 把关键耗时图像处理尽量放到更安全的异步路径中。
- 替换面向开发期的断言式失败处理，增加用户可见错误反馈。

### 5.6 数据结构与代码架构

目标：让当前代码结构可以支撑下一阶段功能扩展，而不是越改越难。

建议：

- 明确评分数据语义，区分：
  - 系统默认评分
  - 用户最终评分
  - 评分来源或备注（如后续需要）
- 逐步清理不再参与展示和业务决策的旧评分字段。
- 把复杂页面状态逐步收拢到更独立的状态对象或 view model。
- 为服务层增加更清晰的职责边界，降低 View 内业务逻辑密度。
- 为数据迁移预留空间，避免未来模型调整成本过高。

### 5.7 测试与工程质量

目标：让迭代过程更安全，减少功能互相影响带来的回归问题。

建议：

- 补充图片处理相关测试用例。
- 补充评分默认值和人工改分的保存测试。
- 补充 `Outfit` 快照一致性测试。
- 补充衣柜筛选、搜索、排序等逻辑测试。
- 建立针对关键核心服务的最小单测覆盖基线。

---

## 6. 优先级排序

### P0：优先立即处理

这些事项会直接影响当前 app 的日常可用性和后续迭代稳定性。

- 单品编辑能力
- 衣柜页搜索、排序、多维筛选
- 评分模型语义收敛
- 关键操作的统一反馈机制
- 图片压缩与展示性能基础优化

### P1：建议尽快推进

这些事项会明显提升产品完整度和用户感知价值。

- 添加衣物支持连续录入
- 搜配增加场景 / 备注
- 历史记录支持“再次使用这套”
- 搜配失效状态提示优化
- 历史记录页信息层级优化

### P2：中期增强项

这些事项更偏体验升级与长期差异化能力。

- 更真实的穿搭预览生成逻辑
- 更丰富的穿搭统计与提醒
- 收藏 / 常穿 / 待穿体系
- 更完整的设计系统
- 更高覆盖率的测试体系

---

## 7. 建议迭代节奏

### 阶段一：先补可用性基础

目标：把 app 从“能跑通”提升到“能长期使用”。

建议包含：

- 单品编辑
- 搜索 / 排序 / 筛选
- 操作反馈优化
- 图片处理与展示基础优化

### 阶段二：增强搜配价值

目标：让搜配从展示功能变成更有决策价值的功能。

建议包含：

- 场景与备注
- 更好的评分呈现
- 历史复用能力
- 更明确的推荐与调整路径

### 阶段三：提高产品差异化

目标：让产品从基础工具走向更有特色的个人穿搭助手。

建议包含：

- 更强的预览能力
- 更完整的数据洞察
- 更成熟的设计表达
- 更高的稳定性与扩展能力

---

## 8. 适合立刻落地的改动项

以下事项适合作为下一轮开发的第一批任务：

- 衣柜页增加搜索框
- 衣柜页增加排序功能
- 单品详情页增加编辑入口
- 添加衣物页增加“保存并继续添加”
- 搜配页增加场景或备注输入
- 历史记录增加“再次使用这套”
- 清理评分数据的旧结构，统一默认分与最终分语义
- 补充图片处理与评分保存链路测试

---

## 9. 推荐的下一步执行方案

建议按以下顺序推进：

1. 先完成 P0 第一批基础能力
2. 再完善搜配与历史页的复用价值
3. 最后投入更重的预览、统计与差异化功能

如果后续继续执行，建议将下一轮开发任务拆成一份更具体的实施清单，例如：

- Sprint 1：衣柜管理增强
- Sprint 2：搜配体验增强
- Sprint 3：记录与复用增强
- Sprint 4：稳定性与架构清理

---

## 10. 结论

当前 `衣柜管家` 已具备良好的 MVP 基础，但下一步不应只做零散功能追加，而应围绕“可用性、复用性、可信度、可维护性”四个核心目标持续推进。

建议优先把管理能力、编辑能力、评分语义和记录复用能力补齐，再进入更高级的预览与智能化优化阶段。
